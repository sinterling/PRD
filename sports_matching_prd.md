# [PRD] 우리 동네 스포츠 용병 모집 및 즉석 매칭 플랫폼 '동네용병(가칭)'

## 1. 5W1H 표
| 항목 | 내용 |
|---|---|
| **Who (대상)** | 팀 스포츠(풋살, 농구 등) 인원이 부족한 팀장 & 혼자서도 팀 운동에 참여하고 싶은 개인 |
| **When (언제)** | 퇴근 후, 주말 등 갑자기 운동 일정이 생기거나 당일 결원이 발생했을 때 |
| **Where (어디서)** | 사용자 현재 위치 또는 지정 위치 반경 5~10km 이내의 로컬 스포츠 시설 |
| **What (무엇을)** | 실시간 용병/게스트 모집 및 즉석 참가 신청 매칭 |
| **Why (왜)** | 인원 부족으로 인한 구장 대여비 낭비 및 경기 취소 방지, 개인의 스포츠 접근성 향상 |
| **How (어떻게)** | LBS(위치 기반 서비스)로 내 주변 모집글을 지도에 시각화하고, 간편 결제로 노쇼 보증금을 걸어 즉시 매칭 |

## 2. 기능 상세 정의
개발자가 즉시 아키텍처를 그릴 수 있도록 핵심 기능과 기술적 고려사항을 정의합니다.

*   **F1. 위치 기반 실시간 매칭 맵 (LBS)**
    *   **설명:** Google Maps / Kakao Maps API를 활용하여 주변 구장의 매칭 현황을 마커로 표시합니다.
    *   **개발 포인트:** PostGIS(PostgreSQL)의 `ST_Distance` 등 공간 쿼리를 활용해 반경(1km/3km/5km) 내 활성 모집글을 필터링합니다. 클라이언트 성능 최적화를 위해 마커 클러스터링을 적용합니다.
*   **F2. 용병 모집글 생성 및 동시성 제어**
    *   **설명:** 종목, 일시, 구장 위치, 필요 인원, 참가비, 요구 실력(하/중/상)을 입력하여 모집글을 생성합니다.
    *   **개발 포인트:** 잔여 인원 카운팅 시 **Redis 분산 락(Redisson)** 또는 **DB 비관적 락(Pessimistic Lock)**을 활용하여 초과 모집(Overbooking)되는 동시성 이슈를 완벽히 차단해야 합니다.
*   **F3. 참가 신청 및 에스크로 간편결제**
    *   **설명:** 노쇼(No-show)를 막기 위해 참가비 또는 보증금을 선결제해야만 신청이 완료됩니다.
    *   **개발 포인트:** Toss Payments 또는 아임포트(PortOne) API를 연동합니다. 결제된 금액은 서버에서 에스크로 형태로 Hold하고, 경기 종료 및 출석 확인 후 호스트에게 정산되거나 게스트에게 환불됩니다.
*   **F4. QR 기반 출석 체크 및 실시간 상태 알림**
    *   **설명:** 경기장 도착 시 호스트 앱의 QR 코드를 게스트가 스캔하여 출석을 인증합니다.
    *   **개발 포인트:** Socket.io 또는 Spring WebFlux를 통해 신청/수락/출석 상태를 즉각 클라이언트에 반영합니다. QR 코드는 보안을 위해 1분 단위로 갱신되는 TOTP(Time-based One-Time Password) 로직을 적용합니다.
*   **F5. 3차원 매너 평가 시스템**
    *   **설명:** 경기 종료 후 실력, 매너, 시간 엄수 여부를 상호 평가하여 '매너 온도'에 반영합니다.
    *   **개발 포인트:** 평가 데이터 처리 시 DB 부하를 줄이기 위해 Message Queue(RabbitMQ/Kafka)를 활용한 비동기 일괄 업데이트를 권장합니다.

## 3. UX Flow

### 호스트(팀장) Flow
1. **메인 홈:** 모집글 작성 플로팅 버튼(FAB) 탭
2. **조건 입력:** 지도에서 구장 선택 후 일시, 필요 인원, 1인당 참가비 세팅
3. **결제 연동:** 정산 받을 계좌 확인 -> [모집 시작]
4. **매칭 진행:** (푸시 알림 수신) 신청자 프로필 및 매너 온도 확인 후 [수락] / [거절]
5. **경기 당일:** 출석용 인증 QR 코드 화면 활성화
6. **종료 및 정산:** 참석자 평가 완료 시 에스크로 계좌에서 정산금 입금 처리

### 게스트(용병) Flow
1. **메인 홈 (맵 뷰):** GPS 기반 현재 위치 주변의 활성화된 모집 마커 탐색
2. **상세 확인:** 마커 탭 후 모집 상세 정보, 호스트 매너 온도, 기존 참여자 프로필 확인
3. **결제/신청:** [신청하기] 탭 -> 간편 결제 바텀 시트 호출 -> 참가비 결제 완료
4. **커뮤니케이션:** 매칭 채팅방 자동 초대 (공지사항 수신)
5. **경기 참여:** 경기장 도착 -> 호스트 앱 QR 스캔하여 [출석 완료] 처리
6. **종료 후:** 호스트 평가 진행 -> 평가 완료 시 보증금 반환 (또는 참가비 확정)

## 4. 기술 스택 (권장)
*   **Frontend (App):** `React Native (Expo)` 또는 `Flutter`
    *   *선정 사유:* iOS/Android 동시 런칭 및 지도 API, GPS 트래킹 등 네이티브 기능 최적화. 상태 관리는 Zustand(React) 또는 Riverpod(Flutter) 사용.
*   **Backend:** `Node.js (NestJS)` 또는 `Spring Boot (Java/Kotlin)`
    *   *선정 사유:* 에스크로 결제 및 인원 차감 등 안정적인 트랜잭션 관리에 유리.
*   **Database & Cache:** `PostgreSQL (PostGIS)`, `Redis`
    *   *선정 사유:* LBS(위치기반) 쿼리를 위해 PostGIS 확장이 필수적이며, 캐싱 및 동시성 제어(분산 락)를 위해 Redis 채택.
*   **Infra & CI/CD:** `AWS (EC2, RDS, ElastiCache, S3)`, `GitHub Actions`, `Docker`
*   **3rd Party API:**
    *   지도: `Kakao Maps API` (국내 구장 데이터 및 주소 변환)
    *   결제: `Toss Payments`
    *   알림: `Firebase Cloud Messaging (FCM)`

## 5. MVP 로드맵 (8주 스프린트)

*   **Sprint 1 (Week 1-2) - 설계 및 인프라 구축**
    *   요구사항 정의 확정, API 명세서(Swagger) 작성.
    *   PostGIS 활용 DB 모델링 및 AWS 클라우드 인프라 프로비저닝.
    *   크로스 플랫폼 앱(RN/Flutter) 프로젝트 초기 세팅.
*   **Sprint 2 (Week 3-4) - 핵심 LBS 매칭 기능**
    *   소셜 로그인(OAuth) 및 JWT 인증 구현.
    *   지도 API 연동 및 현재 위치 기반 마커 표시.
    *   모집글 CRUD 및 거리 기반 필터링(PostGIS) API 연동.
*   **Sprint 3 (Week 5-6) - 결제 및 동시성 제어**
    *   Toss Payments 테스트망 연동 (보증금 결제 프로세스).
    *   Redis 분산 락을 적용한 인원 차감 동시성 제어 로직 구현.
    *   실시간 알림(FCM) 및 안내용 웹소켓 채팅방 생성.
*   **Sprint 4 (Week 7-8) - 출석/평가 및 QA 배포**
    *   QR 스캔 출석 로직 및 매너 온도 평가 시스템 구현.
    *   전체 시나리오 통합 QA 및 버그 픽스.
    *   앱스토어/플레이스토어 심사 등록 및 클로즈 베타(특정 지역 한정) 런칭.

## 6. 리스크 및 대응 전략

| 리스크 요인 (Risk) | 심각도 | 대응 전략 (Mitigation) |
|---|:---:|---|
| **잦은 노쇼(No-show)로 인한 팀장들의 이탈** | High | **100% 선결제 에스크로 시스템** 도입. 경기 시작 전 노쇼 시 보증금 전액 호스트에게 귀속. 누적 노쇼 2회 이상 시 한 달간 서비스 이용 영구 정지(Hard Ban). |
| **인기 구장 초과 모집(Overbooking) 동시성 이슈** | High | 결제 진입 시점에 트랜잭션 **락(Lock)**을 걸고, Redis `INCR/DECR` 원자적 연산을 활용해 정확히 남은 TO만큼만 결제가 이루어지도록 백엔드 로직 강화. |
| **초기 유저풀 부족 (Cold Start Problem)** | Medium | 전국 런칭을 피하고 **특정 밀집 지역(예: 강남구/송파구)과 특정 종목(풋살)으로 한정**하여 베타 오픈. 해당 지역 주요 구장과 제휴하여 오프라인 배너(QR코드) 적극 배치. |
| **GPS 조작을 통한 허위 출석 인증** | Low | 단순 GPS 인증을 폐기하고, 호스트 기기에서 생성된 **동적 QR 코드 스캔 방식**으로 물리적 출석 인증 강제화. |
